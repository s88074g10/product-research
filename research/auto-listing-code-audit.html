作為「會跑、有產出的內部 PoC」合格——作者對各平台 API 的實戰細節打磨得很細。但作為「要長期維護、會真的改動正式後台」的工具，工程衛生不及格：有 2 個必須立刻處理的安全問題，以及全模組共通的 3 個結構性硬傷。

02🔴 必須立刻處理（安全）

嚴重 Yahoo購物/sign_in.py:21–26 ✓ 已親自驗證
Yahoo AES 金鑰 / 簽章 salt 硬編在原始碼
KEYVALUE = os.getenv("YH_KEYVALUE") or "************ 已遮罩 ************"
KEYIV    = os.getenv("YH_KEYIV")    or "******** 已遮罩 ********"
SALTKEY  = os.getenv("YH_SALTKEY")  or "************ 已遮罩 ************"

這組 AES 金鑰 + HMAC salt + token 是 Yahoo 供應商身分的完整鑰匙，等同「上架／改價／改商品」的後台權限。任何拿到這支 .py 的人（外包、離職員工、若曾上傳 git）都能偽造身分。而第 28 行「缺金鑰就報錯」的保護因為有 or "硬編值" 永遠不會觸發，等於 .env 形同虛設。硬編這組與 .env 內實際那組不一樣，疑似舊金鑰，更該確認是否已外流。

動作：① 向 Yahoo 申請輪替（rotate）金鑰；② 移除所有 or "..." fallback，缺 env 直接報錯；③ 確認此檔從未進過 git。
嚴重 Yahoo購物/sign_in.py:114–117, 209–259
簽章疑似用錯 key，靠暴力試 KeyVersion 1~5 遮掩

HMAC 的 key 傳的是 AES 金鑰的 base64 字串，salt key 卻被當訊息串進去——很可能不符 Yahoo 規格。程式之所以要「自動試 5 個 KeyVersion」，很可能就是簽章本身不確定對不對。正確簽章不需要猜。

動作：對照 Yahoo 官方文件確認簽章規格，修對後拿掉暴力迴圈。

03🟠 全模組共通的 3 個結構性問題

A

寫死特定同仁的個人 OneDrive 路徑

C:\Users\ch976\OneDrive\Desktop\Anson\上架素材（三個模組都有）。換人、換機就全掛，可攜性為零、無法交接。應移到 .env（Friday 的 config.py 其實已預留 IMAGE_ROOT 卻沒人用）。

B

無「重複上架防護」（冪等性）

這是批次上架工具最該有、卻三個模組都缺的。失敗後重跑整份 Excel，會把已成功的列再次送出，在後台造成重複商品／重複提案。沒有去重、沒有斷點續傳，Excel 也是全部跑完才一次存檔（中途崩潰結果全失）。

C

大量重複碼 + 一次性測試腳本散落

同樣的 api_get / headers / normalize / upload 在 8+ 支檔案各抄一份，細節還已分歧（reqid 兩種算法、圖片快取 3 種策略、SSD 正規化 3 支不一致）。Yahoo 那邊還有一堆寫死「單一台 ASUS FA707NUG」的探針腳本混在正式工具裡。

05給您的建議（決策層面）

  1. 這工具要正式化還是維持 PoC？若要長期用，請排一次重構：抽共用模組、路徑進設定、檔名改英文、移出 _tmp/。

01🔴 必修

這三項即使一個人用、即使在自己電腦跑，也一樣會出事。以下為建議改法，由作者本人修改。

必修 1sign_in.py:21–26
Yahoo 金鑰硬編在原始碼

原始碼帶著鑰匙，只要這支檔被備份到雲端、隨手分享、或哪天同步到公開空間就外洩。建議：把 os.getenv(...) or "金鑰" 改成只讀 .env、缺則報錯，並確認 .env/.py 沒進過 git、沒同步到公開資料夾。

輪替只在「確定曾外洩」時才需要——單人本機用威脅較低，先移除硬編 + 確認沒外流即可。
